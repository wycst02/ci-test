# RFC 7230: HTTP/1.1 消息语法和路由

**Status**: Standards Track
**Obsoletes**: RFC 2616
**Authors**: R. Fielding, J. Reschke
**Date**: June 2014

## 概述

RFC 7230 定义了 HTTP/1.1 的消息语法、传输机制和路由规则。

## 核心概念

### 消息结构
```
HTTP-message = start-line
               *(header-field CRLF)
               CRLF
               [ message-body ]
```

### 请求消息
```
Request = request-line
           *( header-field CRLF )
           CRLF
           [ message-body ]
```

### 响应消息
```
Response = status-line
            *( header-field CRLF )
            CRLF
            [ message-body ]
```

## HTTP 版本

```
HTTP-Version = "HTTP" "/" 1*DIGIT "." 1*DIGIT
```

示例: HTTP/1.1, HTTP/1.0

## 协议版本规则

- 发送 "HTTP/1.1" 必须兼容 RFC 7230-7235
- 代理不应升级请求版本
- 版本比较是字符串比较

## 统一资源标识符 (URI)

### request-target 形式
```
request-target = origin-form
               / absolute-form
               / authority-form
               / asterisk-form
```

| Form | 用途 | 示例 |
|------|------|------|
| origin-form | 直接服务器请求 | /pub/WWW/TheProject.html |
| absolute-form | 代理请求 | http://www.example.org/pub/WWW |
| authority-form | CONNECT | www.example.org:80 |
| asterisk-form | OPTIONS * | OPTIONS * HTTP/1.1 |

### URI 引用
```
URI-reference = <URI, see [RFC3986], Section 4.1>
absolute-URI  = scheme ":" hier-part [ "?" query ]
relative-ref  = relative-part [ "?" query ] [ "#" fragment ]
```

## 头字段

### 语法
```
header-field = field-name ":" OWS field-value OWS
field-name   = token
field-value = *( field-content / obs-fold )
field-content = field-vchar [ 1*( SP / HTAB / field-vchar / obs-fold ) field-vchar ]
```

### 关键头字段

| Field Name | 描述 |
|------------|------|
| Host | 目标主机 (必需) |
| Content-Length | 消息体长度 |
| Transfer-Encoding | 传输编码 |
| Connection | 连接控制 |
| Trailer | 拖车字段 |

### Connection 头字段
```
Connection = 1#connection-option
connection-option = token
```

### Transfer-Encoding
```
Transfer-Encoding = 1#transfer-coding
transfer-coding = "chunked" / "compress" / "deflate" / "gzip" / transfer-extension
```

### Chunked 编码
```
Chunked-Body = *chunk
               last-chunk
               trailer
               CRLF

chunk          = chunk-size [ chunk-ext ] CRLF
               chunk-data CRLF
chunk-size     = 1*HEX
last-chunk     = 1*("0") [ chunk-ext ] CRLF
```

## 消息体长度

确定消息体长度的规则（按优先级）:

1. 1xx/204/304 响应和 HEAD 请求: 无消息体
2. Transfer-Encoding: chunked
3. Content-Length 头字段
4. 多媒体类型 multipart/byteranges
5. 服务器关闭连接

## 传输管理

### 持久连接
```
Connection: keep-alive
```

HTTP/1.1 默认使用持久连接。

### 管道化
```
Pipeline = 1*( HTTP-request )
```

## 分帧

### 起始行
```
start-line = Request-Line / Status-Line
```

### 异常处理
- 收到空行后跟请求行 → 忽略空行
- 收到无效起始行 → 400 Bad Request
- 版本不匹配 → 505 HTTP Version Not Supported

## 扩展性

### 协议扩展
```
protocol-version = product *( "/" product-version )
```

### 头字段扩展
- 新头字段通过 IANA 注册
- 不认识的头字段通常作为实体头字段处理

## 与 RFC 2616 的主要区别

1. **ABNF 扩展**: 引入 # 操作符定义列表
2. **request-target**: 明确四种形式
3. **Host 头字段**: 必须存在
4. **chunked 编码**: 必须最后应用
5. **连接控制**: 更明确的语义

## 实现建议

1. 始终发送有效的 HTTP 版本
2. 处理不完整的行（容忍应用程序）
3. 正确实现分块传输编码
4. 处理持久连接的关闭
5. 正确计算消息长度
